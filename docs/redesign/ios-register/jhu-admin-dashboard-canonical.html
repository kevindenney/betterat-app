The desktop admin surface — schools as tenants, not as a skin

This is the web surface administrators use to run a school of nursing on BetterAt: roster, cohorts, capability mapping, billing, and the org’s public profile. Sized for laptops and large monitors; mobile-responsive degradation exists but is not the canonical view. Inherits the iOS register’s palette and gravity — system grays for chrome, iOS blue for actions — but trades the phone shell for a sidebar + tab-strip layout.

The tenant on screen is Westbrook University School of Nursing, a placeholder we use across product canonicals. Real institutions are themed at the strap-and-shield level only; product chrome, accent color, and typography stay constant across every tenant.

Frame 2 · Roster

Members · status segment, role chips, ten rows that show the mix

The default view is Active filtered down by role chip; Pending, Rejected, and All sit alongside but rarely lead. Role chips are additive, not exclusive — a member can carry preceptor and evaluator at once, and the row badges show all of them. The Invite button anchors the top-right so it’s reachable without scroll.

Frame 3 · Invitation modal

Invite · tokenized emails, six clear role cards, confirmation only where it counts

The composer treats emails as tokens so paste-from-spreadsheet is the dominant happy path; invalid addresses flag inline rather than failing on submit. Role is a single choice applied to the whole batch (mixed-role batches are vanishingly rare). Only Org Admin carries a ‘requires confirmation’ tag — every other role is one-tap accept.

Frame 4 · Competencies

Competencies · the framework as it’s actually configured

AACN’s ten domains are the spine; everything beneath is the school’s configuration on top. Domain 2 is opened to show evidence, source-link, and the per-row Customize wording affordance. The right rail surfaces what an admin needs to act on — unmapped domains, missing evidence requirements, wording overdue for review.

Mobile-responsive degradation exists — the tab strip collapses into a select, side rails stack under the main column, and the modal goes full-screen at <640px — but the canonical view is desktop. Mobile shots live in the responsive companion canonical.

What carried over from the iOS register

Stage chrome stays warm and serif so this surface reads as part of the same review series. Inside the desktop window the rules are iOS’s: SF Pro across the UI, system grays for chrome, iOS blue as the single owner of action color. Cards are white with the same 0.5px hairline borders and modest two-stop shadow language from Practice Timeline.

Density is the visible difference. A desktop row holds five columns where a phone row held two; the chrome doesn’t have to fight for vertical space, so widgets can breathe and stats can run large.

Schools as tenants

Every institution gets the same product chrome. What changes per tenant: the shield on the org strap, the org name beside it, and the capability customizations they choose under Competencies. iOS blue, sidebar layout, typography, and the entire tab grammar stay constant.

This is deliberate: an admin moving between two BetterAt orgs should never feel like they switched products. The placeholder ‘W’ shield is what every new tenant gets until they upload their own mark.

The onboarding card

New surface in this addendum. Lives on Dashboard only, dismissible with a sticky state per-user, and returns from Help if dismissed. The four progress dots correspond to the four sub-tours: roster, cohorts, capabilities, and invitation/site flow.

Visual gravity is intentionally low — iOS-blue accent on a soft gradient, no shadow stack — so it reads as a friendly utility, not a marketing banner.

The invitation modal

The redesign moves from a one-email-at-a-time form to a tokenized batch composer. The dominant flow is paste-from-spreadsheet; invalid addresses flag inline rather than failing on submit; the ‘send N invitations’ button reflects the live valid count.

Role is one-choice-per-batch. Org Admin is the only role that triggers a second-party confirmation step — every other invitation accepts in a single tap from the email.
