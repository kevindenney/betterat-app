Surface 3 · Topic detail

The unit is the topic

Downwind mark trim. Felix doesn’t subscribe yet, so Subscribe sits in the hero AND docks into the nav once he scrolls past the hero (shown). Body: pinned slot, recent threads, who’s active (loading), related orgs. Demonstrates the per-section loading position.

What makes these siblings

Same chrome, same hero, same cross-reference grammar — three different bodies.

Five structural rules hold across the three: (1) the nav bar is identical — back chevron carrying the originating segment, tiny centred context label, an action slot that’s either empty or carries the docked relationship affordance; (2) the hero scaffolding is the same — 64 px mark, name, descriptor row, meta pellets, relationship action — only the mark shape and the content varies; (3) section headers are the same 11 px small caps with optional See all →; (4) the cross-reference row (xrow) is one component used on all three; (5) the per-section loading position is the same on all three — hero solid first, sections fill progressively.

What differs is the body’s content sections: institutional activity on Org, current-concept + trajectory on Person, pinned + threads on Topic. The brief calls that out explicitly — the list trio differed in the cell, the detail trio differs in the sections. Same rule shape, one nest deeper.

Hero hierarchy

Heights flex; the scaffold is what reads as sibling.

Position taken: the hero scaffolding is identical (mark + name + descriptor + meta + relationship slot) but the heights are not forced equal — the Org hero is taller because the meta row carries three pellets (location, founded, count); the Person hero is shortest (no founded year, no count). Same rule as the list trio: the column rhythm is what reads as repeating, not the literal pixel height.

The 64 px mark is fixed across all three; that’s the strongest visual hinge. The name is always 22 px display semibold. The meta row, when present, always sits beneath the descriptor. The relationship action is always the last row of the hero. If we forced identical heights we’d be adding fake content to the Person hero or cropping the Org meta — both dishonest, both rejected for the same reason the list trio rejected forced-equal cell heights.

The cross-reference triangle

Each surface points to the other two; three is the minimum to show the section.

Org points to members you may know + club forums. Person points to in common (mutual orgs & topics) + public threads. Topic points to active sailors + hosted by. The triangle is symmetric at the component level — same xrow, same chevron, same 36 px mark.

Position taken on sparse data: three is the minimum to render the section. With one or two cross-refs, the row count looks accidental rather than curated — the user can’t tell whether they’re seeing “the most relevant” or “everything we have.” Below three, the section is absent, not shortened. (The Topic Hosted by section is the explicit exception: one host is the common case for a club topic, so the section renders with a single row and reads as definitional, not as a thinned-out list.)

Activity signals across the three

Same row component, three different jobs — the rendering is identical, the labels carry the difference.

Position taken: the activity signal is one component — the three-pellet signal-row — rendered the same way on Org and Topic. The labels carry the differentiation: races this week / entries / next start on Org; threads / sailors / last active in the Topic hero meta. Person’s activity signal is current concept, which uses a different (italic-serif coral-tinted) component — defensible because the practice signal is editorial in a way race counts and thread counts are not.

The position rejected: rendering all three differently to dramatise that they are different kinds of activity. That would be performance of difference, not structural difference. The signals already are different (numbers, dates, names) — rendering them the same lets the user’s eye focus on what the numbers say, not on visual variety.

The “already mine” question, revisited

The list trio answered with right-edge tags; the detail trio answers in the hero.

On the list, relationship is a property of a row — a small tag (Joined / Following / Mutual) at the right edge, no bolding, no tint. The user is scanning many units; the relationship is metadata.

On detail, the user has chosen this one unit. The relationship is now the first thing they need to know after the unit name: am I in this org / do I follow this person / am I subscribed. So it moves up into the hero, takes a full row, and uses different chrome for the two states — filled iOS-blue button when not-yet-mine (the next useful action is to decide), gray pill with green check when already-mine (the decision is made; the affordance is informational).

The two answers don’t contradict — they extend each other. The right-edge tag and the hero pill describe the same state at different sizes for different jobs (scan vs. decide). The cross-reference rows on each detail surface still carry the small tag (You follow, Mutual, You’re a member) because those rows are mini-lists, not heroes; the grammar holds across nesting levels.

Where the relationship affordance lives, exactly

In the hero on Person + Topic; also docks into the nav on scroll. On Org, it’s absent because the user is already in.

Position taken: the primary relationship action lives in the hero, full-width, iOS-blue filled. The hero is where the user is reading and where the decision is happening. The nav bar at top-of-scroll is reserved for the back chevron and context label only.

On scroll past the hero, the action docks into the nav’s trailing slot at smaller size (shown on the Topic surface). This solves the brief’s constraint — Subscribe must coexist with back-to-Discover — by giving them different ends of the bar and different sizes. They never compete for the same slot.

On Org, Felix is already a member; there is nothing to dock, and the trailing slot stays empty. The secondary Calendar button in the hero is the next useful action for a member, not a relationship action.

Empty within a detail surface

Section absent, not section empty.

Position taken: if a section has no content for this unit, the section does not render. Not “No topics yet.”, not “Nothing here.”, not a placeholder card.

Defence: “no topics yet” is a feed-grammar habit — the surface implies a stream, and an empty stream needs explanation. Detail isn’t a stream; it’s a structured view of what is. If an org has no associated topics, that’s a fact about the org, not an absence in the user’s feed. Showing the section empty would manufacture an absence.

Three instances on this pass: Tomás has public threads (section renders); if he didn’t, the section would simply not appear. The Topic detail has one host (renders, single row, definitional); a topic with no host would have no Hosted by section at all. The Org detail’s Up-next signal would shrink to two pellets, not three, if the club had no Wednesday-night race scheduled — same shape, different content density.

Loading and error positions

Hero solid first — sections fill progressively.

Position taken: the hero region is always rendered before any section. Hero data is small (name, descriptor, meta, relationship state) and travels with the tap from the list cell; the user lands on the hero immediately, with no narration line.

Sections fill in as their data resolves. Each section is independent — a slow recommendation compute on Who’s active (shown) doesn’t block the recent-threads list. The loading narration line lives at section scope, not surface scope; it sits between the section header and the skeleton body, in label-tertiary.

Error: if a section fails, the section becomes the canonical inline error using the same <IOSRegisterErrorState /> register at section size. The hero remains solid; the user can still subscribe / follow / open the calendar. Position rejected: full-surface skeleton until everything resolves — would hold the hero hostage to the slowest section, which is the opposite of what the user came to do.

What this trio explicitly is not

  • Not scaled-up cellsThe list cell answers what is this, at a scan. The detail surface answers what is this, why care, how do I relate, what’s here. Different jobs, different shapes — architecture decision #4.
  • Not feed-styleNo reverse-chronological mix of threads, events, follows. The detail surface is a structured view of a unit. The list of recent threads on Topic is the closest to a feed, and even there the unit is the topic, not the chronology.
  • Not thread detailTapping a thread leaves the trio. Thread detail is its own design pass.
  • Not editingOwned units (Felix’s membership at RHKYC, his own subscriptions) don’t expose edit affordances here. Editing is a separate surface that lives in Profile and in work-mode on the Concept detail register.
  • Not moderationNo pin / lock / hide / report affordances on threads. Forum moderation is its own design problem.
  • Not paywallThe Org detail doesn’t carry a membership-purchase flow. If RHKYC’s membership becomes purchasable in-app, that flow gets its own surface.
  • No coral on already-mine stateThe green check on the Member pill is the one exception — coral is reserved for system-attention. You’re a member is not attention; it’s confirmation.
  • No faculty surfaceTomás is a practitioner discoverable by other practitioners. Faculty (Sam, Patricia) gets a different detail register that this pass doesn’t design.
  • No empty mascots, no apology copy, no “coming soon” placeholdersIf a section has no content, it isn’t there. The absences are the design.
