# BetterAt Feature Audit

Canonical QA tracker for the BetterAt product-quality loop.

Last updated: 2026-06-22
Agent phase: Phase 5 re-verification; full Jest green, exact build script still absent

## Scope And Method

This tracker is code-derived. The first pass scanned:
- `app/` route and screen files: 476 non-test files
- `components/`, `hooks/`, `services/`, `providers/`, `api/`
- Supabase edge functions: 73 `supabase/functions/*/index.ts` entries
- Existing tests/specs: 507 files
- Maestro workflows under `.maestro/`

Important finding: the user prompt describes BetterAt as React/Firebase, but the current repo is Expo/React Native + React web using Supabase (`@supabase/supabase-js`), Supabase Edge Functions, Supabase Storage, Postgres/RLS migrations, and some bridge files named `firebaseBridge`. Firebase interactions were not the primary data layer in the scanned app code.

## Existing Commands

From `package.json`:
- `lint`: `npx eslint . --ext .js,.jsx,.ts,.tsx`
- `typecheck`: `npx tsc --noEmit`
- `test`: `jest`
- `build`: no script named exactly `build`; closest existing build command is `build:web`

Per `AGENTS.md`, missing commands are documented rather than invented.

## Status Legend

- `discovered`: feature identified from route/service code
- `needs-test`: user story exists but practical coverage is missing or unclear
- `failing`: verification produced a failure
- `fixed`: clear bug was fixed, pending broader verification
- `verified`: behavior passed relevant verification

## Top Initial Risks

1. Practice timeline active-interest loading can strand the app in `Loading your interest`/`Loading your timeline` states if cache/profile/interest resolution disagree.
2. AI generation workflows depend on parseable model JSON and need defensive parsers per feature.
3. Race and club management write to many tables from large screens, increasing regression risk around permissions and partial saves.
4. Organization admin surfaces span people/cohorts/programs/billing/security and need role-gate coverage.
5. Public/tokenized routes must avoid leaking private data while still loading without signed-in auth.
6. Billing/Stripe/RevenueCat functions are high impact and need smoke/contracts around success/cancel/portal states.
7. Notification/inbox/external connectors depend on cron/webhook behavior that is hard to verify locally.
8. Atlas/map/course/location features depend on platform permissions and map rendering paths.
9. Settings/account destructive flows need confirmation and auth revalidation coverage.
10. Legacy route aliases and duplicated route families make broken navigation easy to introduce.
